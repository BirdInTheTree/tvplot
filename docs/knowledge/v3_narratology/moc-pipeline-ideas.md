# MOC: Идеи для нового пайплайна

Идеи, которые возникают при чтении нарратологии и могут повлиять на архитектуру пайплайна, аналитику или позиционирование продукта.

## Аналитика и продукт

- **Обнаружение незапланированного**: сетевой анализ может показать структуры, которые авторы не проектировали сознательно — то что *вышло*, а не то что *задумывалось*. Это отдельная ценность инструмента: зеркало для showrunner'а. (Moretti 2011 — [[moretti2011/network-regions-reveal-plot-subsystems]])

## Поиск сюжетных линий

- **Подсети вокруг хабов как кандидаты на сюжетные линии**: степенное распределение связей создаёт несколько хабов, каждый — центр своей подсети. A-story = подсеть главного хаба, B-story = второго. Персонажи на пересечении подсетей участвуют в нескольких storylines. Removal experiment как верификация. (Moretti 2011 — [[moretti2011/power-law-distribution-in-character-networks]])

## Метрики

- **Protagonist detection через removal experiment**: протагонист — не тот у кого больше всего связей (degree), а тот, чьё удаление максимально фрагментирует сеть. Можно использовать как контрольный параметр: после того как пайплайн определил главного персонажа, проверить — действительно ли его удаление из графа взаимодействий ломает структуру сильнее всех остальных. Если нет — возможно пайплайн ошибся. (Moretti 2011 — [[moretti2011/protagonist-defined-by-network-stability-not-centrality]])

## Эксперименты

### Removal experiment на GoT
**Цель**: проверить тезис Moretti — что протагонист определяется не degree, а дестабилизацией сети при удалении.

**Данные**: два датасета от Andrew Beveridge:
- Книги: [mathbeveridge/asoiaf](https://github.com/mathbeveridge/asoiaf) — `asoiaf-book1-edges.csv` и др.
- **Сериал HBO**: [mathbeveridge/gameofthrones](https://github.com/mathbeveridge/gameofthrones) — данные по сезонам, по скриптам с genius.com. Взаимодействия: речь, упоминания, совместные сцены.

**Что Beveridge уже сделал**:
- Community detection → 7 сообществ (Ланнистеры, Робб, Бран, Арья, Джон+Стена, Станнис, Дейнерис)
- 6 метрик центральности (degree, weighted degree, eigenvector, PageRank, closeness, betweenness)
- Тирион — лидер по всем кроме betweenness (там Джон Сноу)

**Что НЕ сделано** (наш эксперимент):
- Removal experiments — удаление вершин и замер фрагментации
- Сравнение degree ranking vs. destabilization ranking
- Связь community detection → storyline identification

**План**:
1. Начать с книги (book1) — проще, данные точно чистые
2. Потом повторить по сериалу (season 1) — сравнить результаты

**Отдельный вопрос: устойчивость к методу построения сети.** Книжный датасет построен грубо (ко-окуренция имён в 15 словах текста). Сериальный — точнее (5 типов: прямая речь, упоминания, совместные сцены). Если removal experiment даёт похожие результаты на обоих — значит структура устойчива к методу экстракции. Это критично для нас: наш пайплайн будет строить графы из синопсисов (ещё более грубый метод), и если результат не зависит сильно от способа построения, даже грубая экстракция даст полезный граф.

**Метрики дестабилизации** (при удалении вершины):
1. Число компонент связности — было 1, стало N
2. Доля LCC (largest connected component) — какая часть сети выжила
3. (опционально) Изменение средней длины пути

**Скрипт** (networkx, ~20 строк):
1. Загрузить CSV → взвешенный граф
2. Для каждого персонажа: удалить вершину, посчитать метрики 1-2
3. Отсортировать по убыванию дестабилизации
4. Сравнить с degree ranking — совпадает ли топ?

**Ожидание**: Ned Stark должен быть топ-дестабилизатор (он связывает Winterfell, King's Landing, Wall через Джона). Degree-лидер может быть другим (Тирион по Beveridge).

**Статус**: начат
