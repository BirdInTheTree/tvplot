# Three criteria filter functional from non-functional events

Source: Bal 1988, "Narratology", Ch. 3.2 Events (pp. 155-159)

Not every transition is worth analyzing. Bal (following Barthes) introduces three filters to distinguish **functional events** — those that matter for the fabula's structure:

## 1. Change

The event must describe a process, not a condition. Test: can you rewrite the sentence with a verb of change? "John was cleaning his house. John fell ill and therefore had to stop cleaning" — the word "stop" marks the transition.

## 2. Choice

Functional events **open a fork**, realize a choice, or reveal the result of a choice. If Liz leaves for work and walks straight — non-functional (doesn't affect what follows). If she's hit by a car at the crossing — functional, because the rest of the fabula depends on it.

Key test: **if this hadn't happened, would the rest change?** If no — not a functional event.

## 3. Confrontation

Every functional event has the structure: subject – predicate – (direct) object. Both subject and object must be actors (agents of action). Two actors and one process.

"John kills a fly" — not functional if the fly isn't an actor in the fabula. "Liz writes a letter" — seemingly no confrontation, but a letter presupposes an addressee → can be rewritten as "Liz writes to John" → confrontation exists.

Context decides: a murder can be non-functional if it merely illustrates a character trait without affecting the fabula's development (Bal cites Joyce Carol Oates, "The Man That Turned into a Statue").

## What this means for tvplotlines

Current pass2 instruction says "Each sentence should result in at least one event." This is too coarse. A sentence like "The team discusses the case" or "John arrives home" is not necessarily a functional event. Adding the choice test — *does this open a fork or realize one?* — would filter noise from the event list.
