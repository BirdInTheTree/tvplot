# План чтения статей для autoresearch v3

Мы проектируем autoresearch для переписывания промптов tvplotlines. Прежде чем писать план, нужно прочитать 7 статей. Ниже — что искать в каждой, какие вопросы закрыть, и в каком порядке читать.

## Порядок чтения

Читать в этом порядке — каждая следующая опирается на предыдущие.

---

### 1. GoLLIE (уже в Zotero, помечена `2read`)

**Что это:** Annotation guidelines в промпте улучшают zero-shot information extraction.

**Ссылка:** В Zotero, key `XBVCW78U`. Или: https://arxiv.org/abs/2310.03668

**Зачем нам:** Это прямо про наш случай — мы даём LLM определения (franchise type, plotline, Story DNA) и ожидаем что она по ним извлекает структуру. GoLLIE показывает как правильно формулировать эти определения чтобы extraction работал лучше.

**Что искать:**
- Как они форматируют annotation guidelines? Есть ли у них "шаблон" определения?
- Как измеряют качество? Какие метрики для IE?
- Помогает ли давать негативные примеры ("что NOT является X")?
- Влияет ли порядок определений на качество?

**Какой вопрос закрывает:** как правильно написать секцию DEFINITIONS в наших промптах (Blueprint First). Как определения из глоссария переводить в формат промпта.

---

### 2. Blueprint First, Model Second (Qiu et al., 2025)

**Ссылка:** https://arxiv.org/abs/2508.02721

**Зачем нам:** Формализует то что tvplotlines уже делает (pipeline с code validation), но может подсказать как сделать лучше. Принцип: код решает workflow, LLM только для bounded subtasks.

**Что искать:**
- Как они структурируют "Execution Blueprint"? Есть ли формат/шаблон?
- Какие subtasks отдают LLM, какие — коду? Есть ли критерий разделения?
- Как обрабатывают ошибки LLM? Retry? Fallback?
- Сравнение с подходом "дать LLM всю задачу целиком" — на сколько лучше modular?

**Какой вопрос закрывает:** что из наших промптов нужно вынести в код (validation rules, constraints, ранжирование по весам). Какая архитектура промпта оптимальна.

---

### 3. DSPy (Khattab et al., ICLR 2024)

**Ссылка:** https://arxiv.org/abs/2310.03714

**Зачем нам:** Идея "промпт = компилируемая программа с сигнатурой". Мы не будем использовать DSPy фреймворк напрямую, но концепция composable modules + automatic optimization релевантна.

**Что искать:**
- Формат "signature" — как описывают вход/выход модуля? Можем ли адаптировать для наших Pass?
- Как работает оптимизация инструкций? Что именно оптимизируется — текст, примеры, оба?
- Multi-stage pipeline — как модули связываются? Как результат одного модуля передаётся другому?
- Метрики — как DSPy оценивает качество pipeline?

**Какой вопрос закрывает:** можно ли автоматически подбирать few-shot примеры для каждого Pass. Как формализовать контракт между пассами.

**Внимание:** статья длинная и техническая. Можно ограничиться секциями 1-3 (концепция) + 5 (эксперименты). Детали реализации (секция 4) — по желанию.

---

### 4. MIPRO (EMNLP 2024) — оптимизатор DSPy

**Ссылка:** https://arxiv.org/abs/2406.11695

**Зачем нам:** Конкретный алгоритм оптимизации для multi-stage pipeline. Оптимизирует инструкции + few-shot примеры совместно. Ближе всего к тому что нам нужно.

**Что искать:**
- Как работает "grounded proposal" — LLM видит код программы + данные + traces и предлагает инструкции?
- Сколько прогонов нужно для оптимизации? Какой бюджет?
- Как они решают проблему "оптимизирую один модуль — ломаю другой"?
- Результаты на multi-stage задачах — насколько улучшение vs human-written?

**Какой вопрос закрывает:** сколько будет стоить autoresearch (количество прогонов). Как избежать ситуации "поправил Pass 1 — сломал Pass 2".

**Совет:** читать после DSPy, иначе контекст будет непонятен.

---

### 5. OPRO (DeepMind, 2023)

**Ссылка:** https://arxiv.org/abs/2309.03409

**Зачем нам:** Самый простой подход к оптимизации промптов. LLM видит прошлые варианты + их scores, генерирует новый вариант. Можем использовать для оптимизации формулировок определений (как описать catalyst, crisis и т.д.).

**Что искать:**
- Формат "meta-prompt" — как подаётся история попыток?
- Работает ли на длинных промптах или только на коротких инструкциях?
- Сколько итераций до сходимости?
- Ограничения — где не работает?

**Какой вопрос закрывает:** подходит ли OPRO для оптимизации описаний event functions (catalyst, crisis) или нужно что-то сложнее.

---

### 6. TextGrad (Stanford→Nature, 2024)

**Ссылка:** https://arxiv.org/abs/2406.07496

**Зачем нам:** "Текстовые градиенты" — LLM даёт текстовый feedback на результат, этот feedback используется для улучшения компонентов. Идея: прогнали pipeline → LLM-judge говорит "storyline X выглядит как дубликат Y" → этот feedback корректирует промпт.

**Что искать:**
- Как работает "backpropagation" через текст? Как feedback от конца pipeline доходит до начала?
- Можно ли оптимизировать отдельные секции промпта, а не весь?
- Сравнение с OPRO — когда лучше TextGrad, когда OPRO?
- Примеры оптимизации промптов (не только кода/молекул)

**Какой вопрос закрывает:** можно ли использовать LLM-as-judge не просто как scorer (хорошо/плохо), а как источник конкретных правок ("в определении catalyst не хватает X").

---

### 7. ExtractBench (KDD 2026)

**Ссылка:** https://arxiv.org/abs/2602.12247

**Зачем нам:** Бенчмарк для structured extraction в JSON. Показывает что ширина схемы (количество полей) — главный предиктор провала. Метрики для оценки nested extraction.

**Что искать:**
- Какие метрики они используют для сравнения extracted JSON с ground truth?
- "Semantic array matching" — как сравнивают массивы (у нас events[], plotlines[])?
- Как обрабатывают null и отсутствующие поля?
- Рекомендации по размеру схемы — сколько полей нормально?

**Какой вопрос закрывает:** какие метрики использовать в autoresearch для оценки качества извлечения. Как сравнивать два JSON результата (старый промпт vs новый).

## Опциональное чтение

### Survey of Automatic Prompt Optimization (уже в Zotero, помечена `read`)
Key `KDKFV7TA`. Если не помнишь — пролистать ещё раз чтобы убедиться что мы не упускаем подходы кроме OPRO/TextGrad/MIPRO.

### Narrative Trails (уже в Zotero, 7 дубликатов — нужно мержить)
Coherent storyline extraction via maximum capacity path optimization. Другой подход к нашей задаче — graph-based вместо LLM. Интересно сравнить, но не блокирует autoresearch.

## Чеклист: что нужно закрыть после чтения

После прочтения у нас должны быть ответы на:

- [ ] **Формат определений в промпте** — как правильно писать секцию DEFINITIONS (из GoLLIE)
- [ ] **Что вынести в код** — конкретный список constraints/validations для выноса из промптов (из Blueprint First)
- [ ] **Контракт между пассами** — как формализовать вход/выход каждого Pass (из DSPy)
- [ ] **Стоимость autoresearch** — сколько прогонов нужно, какой бюджет закладывать (из MIPRO)
- [ ] **Метод оптимизации** — OPRO vs TextGrad vs ручной A/B — что подходит для наших промптов
- [ ] **Метрики** — как сравнивать JSON результаты старого и нового промпта (из ExtractBench)
- [ ] **Few-shot стратегия** — один пример (BB) vs несколько vs подбор по franchise type (из DSPy/MIPRO)
